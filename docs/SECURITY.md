# Kage Security Model

Kage is a privacy-preserving memory protocol for autonomous AI agents. This
document describes the trust assumptions, the threat model, and the hardening
checklist for operators.

## 1. Trust assumptions

| Component | Trusted for | Not trusted for |
|-----------|-------------|-----------------|
| **Solana program** (`programs/kage`) | PDA access control; commitment integrity; SP1 Groth16 verification via `sp1-solana` | confidentiality of plaintext memory |
| **SDK encryption engine** (`packages/sdk`) | AES-256-GCM correctness, key derivation, X25519 key agreement | key custody — the caller owns the keypair |
| **Prover service** (`packages/prover-service`) | generating SP1 proofs over well-formed inputs | keeping proof requests secret (operators can see inputs) |
| **API server** (`packages/api`) | routing, rate-limiting, tier enforcement | the LLM responses it relays — these are generated by third-party models |
| **IPFS / Arweave** | availability of encrypted blobs | confidentiality — blobs are stored ciphertext-only |

Users trust their **own keypair** above all. Losing the keypair means losing
access to the vault; no recovery path exists by design.

## 2. Threat model

### In scope

1. **Network attacker** intercepting API ↔ prover ↔ client traffic.
   *Mitigation*: TLS everywhere; `x-api-key` between API and prover.

2. **Malicious peer agent** trying to read another agent's vault.
   *Mitigation*: Solana program enforces `has_one = owner` + `Signer<'info>`
   on every write path. PDAs derive from the owner pubkey, so forging a
   vault requires forging a signature.

3. **Storage provider** trying to read memory contents.
   *Mitigation*: memory is encrypted client-side with AES-256-GCM before
   upload. The storage provider only sees ciphertext + a content hash.

4. **Replay / forged ZK proofs** on reputation or task proofs.
   *Mitigation*: SP1 Groth16 verification on-chain via the `sp1-solana` crate
   (~200K CU). The verifying key is committed to the proof and checked at
   instruction time ([programs/kage/src/instructions/verify_proof.rs](../programs/kage/src/instructions/verify_proof.rs)).

5. **Credential forgery** for DID claims between agents.
   *Mitigation*: credentials are signed with Ed25519 using the agent's
   keypair. `DIDEngine.verifyCredential` rejects tampered claims and
   forged signatures — see [tests/src/did.test.ts](../tests/src/did.test.ts).

6. **Rate-limit DoS** against the prover service.
   *Mitigation*: `tower_governor` limits per-IP burst + sustained requests.
   Configurable via `PROVER_RATE_PER_SECOND` / `PROVER_RATE_BURST`.

### Out of scope

- **Side-channel attacks** on the client machine (keylogging, cold boot, …).
- **LLM prompt injection** — the agent will act on any instruction it accepts
  as user input. Operators must filter input at the application layer.
- **Solana consensus failures** — if Solana itself is compromised, vault
  access control fails.
- **Quantum adversaries** — Ed25519 and X25519 are not post-quantum secure.

## 3. Key custody

Kage uses several key types. Each has a different custody recommendation:

| Key | Holder | Recommendation |
|-----|--------|----------------|
| Agent Solana keypair | agent operator | hardware wallet in prod; never commit to git |
| X25519 messaging key | derived from Solana keypair | same as above |
| DID signing key | agent operator | same as above |
| Vault encryption master | derived client-side from agent keypair | same as above — rotation invalidates old memories |
| `PROVER_API_KEY` | API operator | secrets manager (AWS Secrets Manager, Vault, 1Password) |
| Succinct Network key (`NETWORK_PRIVATE_KEY`) | prover operator | dedicated low-balance wallet; rotate on leak |
| Program upgrade authority | deployer | hardware wallet; consider a multisig |

**Never** commit any of these to git. `.gitignore` already excludes `.env`,
`agent-keypair.json`, and `devnet-new-wallet.json` — verify before every
push that `git status` shows no staged secrets.

## 4. Operational hardening

- **TLS**: terminate HTTPS in front of the API and WebSocket endpoints.
  Public WebSocket messages are encrypted (X25519 key agreement) but the
  envelope is still observable without TLS.
- **CORS**: the API currently uses `cors({ origin: "*" })`. Tighten to
  known frontends before mainnet ([packages/api/src/index.ts](../packages/api/src/index.ts)).
- **Prover isolation**: the prover service should not be reachable from the
  public internet. Put it behind an internal VPC / firewall and let only
  the API host reach it.
- **Auth enforcement**: in production, set both `PROVER_ENFORCE_AUTH=true`
  and `PROVER_API_KEY=<random-32-bytes>`. The service refuses to start
  otherwise when `network` prover mode is active.
- **Rate limits**: default 10 rps sustained / 30 burst per IP. Scale with
  expected load; watch 429 rates on dashboards.
- **Keypair rotation**: if a server-side keypair leaks, rotate the
  `SOLANA_PRIVATE_KEY` and migrate vaults by re-encrypting memories under
  the new key (manual, requires access to the old key).
- **Secret logging**: never log `x-api-key`, `SOLANA_PRIVATE_KEY`, or raw
  encryption keys. The prover service redacts `x-api-key` automatically.

## 5. Incident response

1. **Suspected key compromise** → rotate immediately, revoke all on-chain
   access grants from the affected vault, generate a new keypair, migrate
   memories.
2. **Prover service outage** → API `/health` shows `zk.proverConnected=false`.
   Commits fall back to hash-only mode; no proof is generated. Alert on
   this transition.
3. **Rate-limit storm** → inspect `tower_http::trace` logs for offender IPs;
   temporarily lower `PROVER_RATE_PER_SECOND` or block at the firewall.
4. **On-chain exploit** → halt API writes, coordinate program upgrade via
   the upgrade authority, disclose to users.

## 6. Reporting a vulnerability

Please **do not** open a public issue for security reports. Email
`security@kage.example` (placeholder) with a minimal reproducer. We will
acknowledge within 48 hours.
